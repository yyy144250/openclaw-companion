## 跨窗口通信重构：从双 WebSocket 改为事件桥接

### 问题
之前为了解决 chat 窗口状态不同步，让每个窗口各自建立 WebSocket 连接。用户指出这样浪费资源。

### 解决方案
引入 `src/services/eventBridge.ts`，利用 Tauri 的 `emit/listen` 跨窗口事件系统：
- **mascot 窗口**：拥有唯一的 WebSocket 连接，通过事件广播状态和消息给 chat
- **chat 窗口**：不建立 WebSocket，通过事件接收状态/消息，发送消息时通知 mascot 代发

### 事件协议
- mascot → chat: `ws:status`、`ws:chat_response`、`ws:live2d_action`
- chat → mascot: `ws:send_chat`、`ws:connect`、`ws:disconnect`、`ws:status_sync`

### 修改文件
- **新增** `src/services/eventBridge.ts` — 事件桥接层
- **改动** `src/App.tsx` — mascot 独享 WS，chat 监听事件
- **改动** `src/components/ChatInput.tsx` — chat 窗口通过事件桥发消息
- **改动** `src/components/SettingsPanel.tsx` — chat 窗口通过事件桥连接/断开

## 修复：mascot 透明区域遮挡 chat 窗口

### 问题
mascot 窗口 800×750 + `alwaysOnTop: true`，即使透明区域也会拦截 chat 窗口的鼠标点击。

### 解决方案
在 `lib.rs` 中做 alwaysOnTop 动态切换：
- **chat 打开时**：mascot 取消 `alwaysOnTop`，chat 设置 `always_on_top(true)`
- **chat 关闭时**：恢复 mascot 的 `alwaysOnTop`
- macOS 上 `set_ignore_cursor_events` + mousemove 像素检测方案不可行（设置 ignore 后收不到 mousemove）

## 最终连接架构

事件桥方案（eventBridge.ts）在实际运行中存在跨窗口事件不可靠的问题，**最终回退为每个窗口各自建立 WebSocket 连接**：
- 简单可靠，chat 窗口直接操作 wsService
- TTS 仅在 mascot 窗口播放，避免重复
- eventBridge.ts 保留未删除，后续可再优化
